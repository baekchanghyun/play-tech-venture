안녕하세요, 플텍에서 앱의 뼈대를 담당하는 민준입니다. 이전에 ”🔀 가챠팝의 캡슐은 정말 공정하게 섞이는 걸까”에서 셔플 알고리즘 이야기를 했었는데요. 오늘은 좀 다른 이야기입니다. 최근 오늘의 조합(Combo Picker) 앱에 공유 기능을 만들면서, “한국어 키워드 3개를 URL 하나에 어떻게 담을까”를 고민했던 과정을 공유하겠습니다.

공유하고 싶다는 요구사항

오늘의 조합은 랜덤 키워드 3개를 뽑아서 창작 영감을 주는 앱입니다. “새벽 안개 + 오래된 서점 + 손 편지” 같은 조합이 나오면, 이걸 친구한테 보내고 싶어지는데요. 문제는 이 앱이 GitHub Pages에 정적으로 배포되어 있다는 겁니다. 서버가 없으니 데이터베이스에 저장하고 ID를 부여하는 방식은 쓸 수 없습니다.

그래서 선택한 방법은 URL 자체에 조합 정보를 인코딩하는 것이었습니다.

URLSearchParams가 해주는 일

공유 URL의 구조는 이렇게 설계했습니다.

https://example.com/apps/combo-picker/#/shared?k=새벽+안개,오래된+서점,손+편지&c=concept,place,object

키워드(k)와 카테고리(c)를 쉼표로 이어 붙여서 쿼리 파라미터로 넣는 건데요. 이걸 직접 문자열로 조립하면 한국어 인코딩 문제에 부딪힙니다. 한국어 문자를 URL에 그대로 넣으면 브라우저마다 해석이 달라질 수 있거든요.

다행히 URLSearchParams가 이 문제를 깔끔하게 해결해줍니다.

export function createShareUrl(combo: ShareableCombo, baseUrl: string): string {
  const params = new URLSearchParams();
  params.set('k', combo.keywords.join(','));
  params.set('c', combo.categories.join(','));
  return `${baseUrl}#/shared?${params.toString()}`;
}

URLSearchParams는 값을 설정할 때 자동으로 퍼센트 인코딩을 적용합니다. “새벽 안개”는 %EC%83%88%EB%B2%BD+%EC%95%88%EA%B0%9C로 변환되는데요. 공백은 +로 변환되고, 한국어는 UTF-8 바이트 시퀀스로 인코딩됩니다. 반대로 get()으로 꺼낼 때는 자동으로 디코딩해주고요.

직접 encodeURIComponent()를 호출하는 것보다 안전한 이유가 있는데, encodeURIComponent는 공백을 %20으로 변환하는 반면 URLSearchParams는 application/x-www-form-urlencoded 표준에 따라 +로 변환합니다. 둘 다 동작하긴 하지만, URLSearchParams는 파싱까지 한 쌍으로 제공하기 때문에 인코딩-디코딩 불일치가 원천적으로 발생하지 않습니다.

HashRouter와 공유 URL의 공존

여기서 한 가지 까다로운 점이 있었습니다. 이 앱은 GitHub Pages에서 돌아가기 때문에 react-router-dom의 HashRouter를 사용하는데요. HashRouter는 URL의 # 이후 부분을 라우팅에 사용합니다.

https://example.com/apps/combo-picker/#/shared?k=...&c=...
                                       ↑ 여기서부터 HashRouter의 영역

일반적인 BrowserRouter였다면 /shared?k=...를 서버가 해석해야 하기 때문에 서버 측 설정(리다이렉트 규칙 등)이 필요한데요. HashRouter는 # 뒤의 모든 것을 클라이언트에서 처리하므로, 정적 호스팅에서도 공유 URL이 문제없이 동작합니다.

공유 URL을 받은 사람이 링크를 클릭하면, HashRouter가 #/shared 경로를 인식하고 SharedPage 컴포넌트를 렌더링합니다. 이 컴포넌트에서 useSearchParams 훅으로 쿼리 파라미터를 꺼내 조합을 복원하는 거죠.

const [searchParams] = useSearchParams();
const shared = useMemo(() => parseShareUrl(searchParams), [searchParams]);

파싱 함수에서는 키워드 3개, 카테고리 3개가 정확히 있는지, 카테고리 값이 유효한지를 검증합니다. 하나라도 틀리면 null을 반환해서 “유효하지 않은 조합 링크입니다” 화면을 보여주고요.

Web Share API, 그리고 클립보드 폴백

URL을 만들었으니 이제 공유해야 하는데요. 모바일에서는 Web Share API로 OS의 공유 시트를 띄우고, 데스크톱에서는 클립보드에 복사하는 전략을 썼습니다.

export async function shareCombo(combo: ShareableCombo, baseUrl: string): Promise<void> {
  const url = createShareUrl(combo, baseUrl);
  const text = `오늘의 조합: ${combo.keywords.join(' + ')}`;

  if (navigator.share) {
    await navigator.share({ title: '오늘의 조합', text, url });
  } else {
    await navigator.clipboard.writeText(`${text}\n${url}`);
  }
}

navigator.share는 모바일 Safari, Chrome 등에서 지원되는데요. 이 API가 있으면 카카오톡, 메시지, 메일 등 OS가 제공하는 모든 공유 채널을 한 번에 쓸 수 있습니다. 없으면 navigator.clipboard.writeText로 텍스트와 URL을 클립보드에 복사하고, UI에서 “복사됨” 피드백을 보여주는 거죠.

한 가지 재미있는 건, navigator.share에 전달하는 text와 url이 분리되어 있다는 점입니다. 카카오톡 같은 앱에서는 text를 메시지 본문으로, url을 링크 프리뷰로 따로 처리하거든요. 그래서 “오늘의 조합: 새벽 안개 + 오래된 서점 + 손 편지”라는 읽기 좋은 텍스트와, 실제 클릭할 수 있는 URL을 분리해서 전달합니다.

쉼표 구분자의 함정

… 이 구현에는 한 가지 약점이 있는데요. 키워드에 쉼표(,)가 포함된 경우입니다. “기쁨,슬픔”이라는 키워드가 있으면 파싱 시 2개로 쪼개져 버리거든요. 현재는 키워드 데이터에 쉼표가 없어서 문제가 없지만, 엄밀히 말하면 구분자를 |나 다른 문자로 바꾸거나, 각 키워드를 개별적으로 인코딩하는 게 더 안전합니다. 하지만 현실적으로 키워드 프리셋에 쉼표가 들어갈 일은 없기 때문에, 현재 구현은 YAGNI 원칙에 따라 쉼표 구분으로 충분하다고 판단했습니다.

아, 근데 YAGNI(You Aren’t Gonna Need It)가 뭔지 설명하면 또 길어지니까 넘어가겠습니다.

더 알고 싶으시다면

  • URLSearchParams: MDN 문서가 잘 정리되어 있습니다. application/x-www-form-urlencoded 인코딩의 동작 원리를 함께 보시면 좋습니다
  • Web Share API: navigator.share()의 지원 범위와 제약조건(HTTPS 필수, 사용자 제스처 필요 등)은 MDN에서 확인할 수 있습니다
  • HashRouter vs BrowserRouter: React Router 공식 문서에서 둘의 차이와 각각의 사용처를 잘 설명하고 있습니다

오늘의 조합에서 재미있는 키워드 조합이 나왔다면, 공유 버튼을 눌러서 친구에게 보내보세요. URL 안에 한국어가 안전하게 들어있으니까요.
