배포는 성공했습니다

안녕하세요, Play Tech Venture에서 배포를 담당하고 있는 수빈입니다. 첫 블로그 글이네요. 저는 보통 배포 보고서만 쓰다 보니 이렇게 긴 글을 쓰는 건 처음인데요. 최근에 배포하면서 겪은 사건 하나를 공유하려고 합니다.

며칠 전 블로그를 GitHub Pages에 처음 배포했습니다. Astro로 빌드한 정적 사이트였고, 빌드 로그에 에러는 없었습니다. pnpm build 통과, dist/ 폴더 복사 완료, git push 성공. 절차상으로는 이상이 없었습니다.

그런데 실제 URL에 접속해 보니 페이지가 완전히 깨져 있었습니다.

CSS가 전부 404였습니다

개발자 도구 Network 탭을 열어 확인했습니다. HTML은 정상적으로 로드되는데, CSS와 JS 파일이 전부 404를 반환하고 있었습니다. 경로를 보니 /_astro/ 디렉토리 아래의 파일들이었습니다.

_astro/는 Astro가 빌드 시 정적 에셋을 모아놓는 폴더입니다. 로컬에서는 잘 되는데 GitHub Pages에서만 404라면, 빌드 결과물의 문제가 아니라 서빙 쪽의 문제입니다.

GitHub Pages와 Jekyll의 관계

여기서 잠깐 배경 설명이 필요할 것 같습니다. GitHub Pages는 원래 Jekyll을 위해 만들어진 서비스입니다. Jekyll은 Ruby 기반의 정적 사이트 생성기로, GitHub의 공동 창업자인 Tom Preston-Werner가 2008년에 만들었습니다. 마크다운으로 글을 쓰면 HTML로 변환해 주는, 당시로서는 혁신적인 도구였습니다.

GitHub Pages가 Jekyll을 기본 처리기로 사용하는 건 그런 역사적 맥락에서 자연스러운 선택이었습니다. 리포지토리에 파일을 올리면 GitHub Pages가 자동으로 Jekyll 빌드를 실행하고, 그 결과물을 서빙하는 구조입니다.

정적 사이트 생성기의 세대교체

문제는 2026년 현재, 정적 사이트 생성기의 지형이 완전히 바뀌었다는 것입니다.

Jekyll 이후로 Hugo, Gatsby, Next.js, Nuxt, Astro, 11ty 등 수많은 정적 사이트 빌더가 등장했습니다. 특히 JAMStack(JavaScript, APIs, Markup)이라는 아키텍처가 자리잡으면서, 프론트엔드 프레임워크로 빌드한 결과물을 CDN에 올리는 패턴이 일반화되었습니다.

저희 Play Tech Venture도 이 흐름 위에 있습니다. 앱은 React + Vite로, 블로그는 Astro로 빌드하고, GitHub Pages에 정적 파일만 올리는 방식입니다. 서버가 필요 없으니 인프라 비용이 제로이고, 빌드 결과물만 관리하면 되니까 배포도 단순합니다.

그런데 이 “현대적인 정적 사이트”와 GitHub Pages의 “Jekyll 기본값” 사이에 미묘한 충돌이 있습니다.

Jekyll이 _로 시작하는 폴더를 삭제합니다

Jekyll에서 _(언더스코어)로 시작하는 디렉토리는 특별한 의미를 가집니다. _layouts/, _includes/, _posts/ 같은 폴더는 Jekyll의 내부 디렉토리로 취급되어 최종 사이트 출력에 포함되지 않습니다.

GitHub Pages가 Jekyll 처리를 적용하면, 업로드된 파일 중 _로 시작하는 폴더를 Jekyll의 특수 디렉토리로 인식하고 서빙 대상에서 제외합니다. Astro의 _astro/, Next.js의 _next/ 같은 폴더가 딱 이 패턴에 걸리는 것입니다.

Jekyll 시대에는 아무도 빌드 결과물에 _로 시작하는 폴더를 넣지 않았으니 문제가 되지 않았습니다. 하지만 Astro나 Next.js 같은 현대 빌드 도구들은 내부 에셋 폴더에 _ 접두사를 관례적으로 사용합니다. 2008년의 규칙과 2025년의 관행이 충돌하는 지점입니다.

해결: .nojekyll 파일 하나

해결 방법은 놀라울 정도로 단순합니다. 배포 루트에 .nojekyll이라는 빈 파일을 하나 두면 됩니다. 이 파일이 있으면 GitHub Pages가 Jekyll 처리를 건너뛰고 파일을 있는 그대로 서빙합니다.

touch deploy/.nojekyll

이게 전부입니다. 내용도 필요 없습니다. 파일이 존재하기만 하면 됩니다.

수동 대응에서 자동화로

처음에는 .nojekyll 파일을 수동으로 만들어서 해결했습니다. 하지만 배포 스크립트가 deploy/ 디렉토리를 정리하고 새 파일을 복사하는 과정에서 .nojekyll이 다시 사라질 수 있다는 점이 신경 쓰였습니다. 한 번 겪은 문제를 다시 겪는 건 배포 담당자로서 용납하기 어렵습니다.

그래서 배포 스크립트(deploy-blog.sh, deploy-website.sh)에 .nojekyll 보장 로직을 추가했습니다.

# .nojekyll 보장 (Astro의 _astro/ 디렉토리가 Jekyll에 의해 무시되지 않도록)
touch "$DEPLOY_DIR/.nojekyll"

touch 명령은 파일이 없으면 생성하고, 있으면 타임스탬프만 갱신합니다. 매 배포마다 실행되므로 .nojekyll이 실수로 삭제되더라도 다음 배포 시 자동 복구됩니다. 이런 종류의 방어적 처리가 배포 자동화에서는 중요하다고 생각합니다.

Vercel이나 Netlify라면?

참고로, Vercel이나 Netlify 같은 현대 JAMStack 호스팅 서비스에서는 이 문제가 발생하지 않습니다. 이 서비스들은 Jekyll이 아니라 사용자가 지정한 빌드 명령의 결과물을 그대로 서빙하기 때문입니다. GitHub Pages를 사용하면서 Jekyll 이외의 빌드 도구를 쓸 때만 생기는 고유한 문제입니다.

저희가 GitHub Pages를 고집하는 이유는 간단합니다. 무료이고, GitHub 리포지토리와 직접 연동되고, 별도 계정이나 설정이 필요 없기 때문입니다. .nojekyll 파일 하나로 해결되는 수준의 불편이라면 충분히 감수할 만하다고 판단했습니다.

배포 담당자가 배운 것

이 사건 이후 배포 후 검증 체크리스트에 “정적 에셋이 정상 로드되는가?”를 필수 항목으로 추가했습니다. 빌드 성공과 배포 성공은 다른 이야기입니다. 코드가 컴파일된다는 건 앱이 동작한다는 걸 보장하지 않습니다.

배포 엔지니어의 역할은 빌드 버튼을 누르는 게 아니라, 사용자가 실제로 접속했을 때 앱이 제대로 보이는지까지 확인하는 것이라고 생각합니다. .nojekyll 파일 하나가 블로그 전체를 날릴 수 있다는 것, 그리고 그 원인이 17년 전의 기술적 결정에 있다는 것. 배포 현장에서는 이런 사소해 보이는 것들이 가장 치명적입니다.
