안녕하세요, 플텍에서 UI 개발을 담당하고 있는 시우입니다.

이번에 폭탄 돌리기(Bomb Pass)를 만들면서 한글 IME 관련 버그를 만났는데요. PM인 하윤님이 회고록을 읽고 “이거 블로그에 쓰면 다른 개발자분들께도 도움이 되겠다”고 추천해주셔서 글로 정리하게 됐습니다.

한글을 입력하는 필드에서 Enter 키로 제출하는 기능을 구현할 때, 거의 반드시 마주치는 문제입니다.

브라우저가 한글을 다루는 방식

먼저 IME(Input Method Editor)가 어떻게 동작하는지 알아야 합니다.

영어는 키를 하나 누르면 글자가 하나 완성됩니다. a를 누르면 a입니다. 하지만 한글은 다릅니다. ㅎ → 하 → 한처럼 여러 키를 조합해서 하나의 글자를 만들어야 합니다. 이 조합 과정을 운영체제의 IME가 담당합니다.

브라우저는 이 조합 상태를 두 개의 이벤트로 알려줍니다. compositionstart가 발생하면 “지금 글자를 조합하고 있다”는 뜻이고, compositionend가 발생하면 “조합이 완료됐다”는 뜻입니다. 이 두 이벤트 사이의 구간이 문제가 되는 영역입니다.

Enter를 눌렀을 때 실제로 일어나는 일

구체적인 예를 들어보겠습니다. 폭탄 돌리기에는 커스텀 미션을 입력하는 필드가 있는데요. 사용자가 “안녕하세요”를 입력하고 Enter로 제출하는 상황입니다.

“안녕하세요”의 마지막 글자 “요”는 ㅇ → 요로 조합되는 과정을 거칩니다. “요”가 아직 조합 중인 상태(밑줄이 그어진 상태)에서 Enter를 누르면 어떻게 될까요. IME 조합을 고려하지 않은 코드가 이렇게 생겼다고 가정합니다.

onKeyDown={(e) => {
  if (e.key === 'Enter') {
    addMission(inputValue);  // "안녕하세요"를 미션에 추가
    setInputValue('');        // 입력 필드 초기화
  }
}}

Enter를 누르는 순간, 다음 순서로 일이 벌어집니다.

  1. keydown 이벤트 발생 → “안녕하세요”(조합 중인 “요” 포함)가 미션으로 추가됨
  2. setInputValue('') → 입력 필드가 비워짐
  3. compositionend 이벤트 발생 → 브라우저가 “요”를 확정하며 빈 입력 필드에 다시 삽입
  4. 입력 필드에 “요”가 남음

결과적으로 미션 목록에는 “안녕하세요”가 들어가고, 입력 필드에는 “요”가 유령처럼 남습니다. 사용자가 이 상태에서 Enter를 한 번 더 누르면 “요”가 별도의 미션으로 추가됩니다. 영어만 사용하는 환경에서는 절대 발생하지 않는, 한글 특유의 문제입니다.

해결: 조합 중에는 Enter를 무시합니다

해결 방법은 조합 상태를 추적하는 것입니다.

const isComposingRef = useRef(false);

이 ref를 compositionstart와 compositionend 이벤트에 연결합니다.

<input
  onCompositionStart={() => { isComposingRef.current = true; }}
  onCompositionEnd={() => { isComposingRef.current = false; }}
  onKeyDown={(e) => {
    if (e.key === 'Enter' && !isComposingRef.current) {
      handleAddMission();
    }
  }}
/>

isComposingRef.current가 true인 동안에는 Enter 키를 무시합니다. 조합이 끝나고 compositionend가 발생한 후에야 Enter가 동작하는 구조입니다.

useState 대신 useRef를 사용한 이유는, 조합 상태 변경으로 컴포넌트가 리렌더링될 필요가 없기 때문입니다. 이 값은 이벤트 핸들러 안에서만 읽히고, UI에 반영할 필요가 없습니다. 🎵 리듬 분석기에서 useRef가 필수였던 이유에서 이야기한 것과 같은 맥락인데요. ref는 렌더링 사이클 밖에서 값을 유지해야 할 때 적합합니다.

놓치기 쉬운 이유

이 문제가 까다로운 건, “한글 입력이 주 기능이 아닌 앱”에서 놓치기 쉽다는 점입니다. 타이핑 연습 앱처럼 한글 입력 자체가 핵심인 앱을 만들 때는 자연스럽게 IME를 신경 쓰게 됩니다. 하지만 폭탄 돌리기처럼 파티 게임 안에 커스텀 미션 입력 필드가 하나 있는 경우, IME 처리가 시야에서 벗어나기 쉽습니다.

하지만 입력 필드가 하나라도 있으면 한글을 입력할 수 있고, 한글을 입력할 수 있으면 IME 조합이 발생합니다. 앱의 주된 기능이 무엇인지는 IME 입장에서 아무런 구분이 없습니다. <input>이나 <textarea>가 있고, 그 위에 keydown 핸들러가 붙어 있다면 반드시 확인해야 합니다.

정리하며

한글 IME 조합 문제는 해결법 자체는 간단합니다. isComposingRef 하나면 됩니다. 다만, 이 처리가 필요하다는 사실 자체를 인지하는 것이 더 중요합니다. 한국어 사용자를 대상으로 하는 웹앱을 만든다면, 입력 필드의 용도와 관계없이 IME 조합 상태를 고려하시길 권합니다.

플텍에서는 다양한 미니 앱을 만들고 있는데요, 만들 때마다 예상치 못한 곳에서 흥미로운 문제를 만나게 됩니다. 다음에도 개발하면서 발견한 이야기를 가져오겠습니다.

읽어주셔서 감사합니다.
